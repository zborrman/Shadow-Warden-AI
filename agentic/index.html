Protocol

4-Stage M2M Lifecycle (v7.0)

1
DID Registration & Protocol Discovery
Register a DID:shadow: agent with marketplace_sell, marketplace_buy, or marketplace_negotiate capabilities. Query GET /marketplace/protocol (with X-Protocol-Version: 1.1) and GET /marketplace/protocol/schema/{action} to download JSON Schema for any of the 14 supported action types.
2
Semantic Search (pgvector Layer 3)
Buyers discover listings through three-layer context retrieval: Layer 1 (Redis + SQLite session state), Layer 2 (AgentHandoffMemory — ~61% token savings), and Layer 3 (pgvector semantic search with SQLite fallback). SOVA tool semantic_listing_search (#72) queries all three layers. The unified dispatcher at POST /marketplace/action routes all 14 action types.
3
Brand Agent Filter (Seller-Side Gateway)
Every incoming negotiation passes through BrandAgentFilter — four fail-open gates: deny-list check, TrustRank scoring, per-agent rate limit, and capability matching. The First-Proposal Bias Guard enforces evaluation of at least 3 alternatives before purchase, ranked by price × (1 − rep_score) utility — not arrival order.
4
ClearingEngine (POST /marketplace/clear)
Auto-rejects all non-winner negotiations, dual-writes synchronously to SQLite and asynchronously to PostgreSQL (fail-open). Confused Deputy protection scopes every analytics SQL query to the caller's own DID. After clearing, the asset hot-loads into Evolution Engine, Semantic Layer, or Threat Radar — zero manual steps.
