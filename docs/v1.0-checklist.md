# Shadow Warden AI — v1.0 Release Checklist

> **Target:** Private Enterprise installations (US / EU)
> **Status gate:** All items must be ✅ before tagging `v1.0`.

## 1. Security & Filtering (Core Security)

### Detection Pipeline (5 layers)

- [x] **Layer 0 — ObfuscationDecoder** (`warden/obfuscation.py`)
  Decodes Base64 blobs, hex pairs, ROT-13, and Unicode homoglyphs
  (Cyrillic/Greek → Latin) before any downstream analysis.
  Fires at stage 0 so encoded jailbreaks cannot bypass regex or ML layers.

- [x] **Layer 1 — SecretRedactor** (`warden/secret_redactor.py`)
  15+ compiled regex patterns: OpenAI / Anthropic / HuggingFace keys,
  AWS credentials, GitHub tokens, private keys, credit-card numbers,
  SSNs, IBANs, email addresses, IPv4 addresses, JWTs.
  Replaces matches with `[REDACTED:<kind>]` tokens in `filtered_content`.

- [x] **Layer 2 — SemanticGuard** (`warden/semantic_guard.py`)
  Rule-based engine covering OWASP LLM Top-10:
  - **LLM01** — Indirect Prompt Injection (separator attacks, role overrides)
  - **LLM05** — Insecure Output (XSS, command injection, SSRF, path traversal)
  - **LLM06** — Excessive Agency (autonomous action claims, privilege escalation)

- [x] **Layer 3 — BrainSemanticGuard** (`warden/brain/semantic.py`)
  all-MiniLM-L6-v2 cosine similarity against a curated jailbreak corpus.
  Non-blocking: runs in a `ThreadPoolExecutor` via `check_async()`.
  Singleton loaded at startup via `@lru_cache(maxsize=1)`.

- [x] **Layer 4 — EvolutionEngine** (`warden/brain/evolve.py`)
  Background loop powered by Claude Opus. On HIGH/BLOCK events, generates
  new detection rules via streaming + adaptive thinking and hot-reloads
  them into BrainSemanticGuard via `add_examples()`.
  Gracefully disabled when `ANTHROPIC_API_KEY` is empty (air-gapped mode).

### Risk Escalation

- [x] **Compound Risk** — 3 or more `MEDIUM` signals in a single request
  are escalated to `HIGH` before the allow/block decision.
  Implemented in `SemanticGuard.analyse()` (`warden/semantic_guard.py`).

## 2. Infrastructure & mTLS

- [x] **Internal CA** (`scripts/gen_certs.sh`)
  Generates a 4096-bit Root CA (10-year) and 2048-bit leaf certificates
  with 90-day validity for all services: `warden`, `analytics`, `admin`,
  `proxy-client`, `analytics-client`, `app-client`, `admin-client`.
  Warden server cert includes Kubernetes SANs
  (`<release>-warden.<namespace>.svc.cluster.local`).

- [x] **CRL generation & revocation**
  `gen_certs.sh revoke <name>` marks the cert revoked in the OpenSSL CA
  database and regenerates `certs/ca.crl` (30-day validity).
  When `kubectl` + `helm` are in PATH, the command automatically:
  1. Pushes the updated Secret to Kubernetes (`kube-apply`)
  2. Removes the revoked CN from `MTLS_ALLOWED_CNS` via `helm upgrade`
  3. Restarts the warden deployment (`kubectl rollout restart`)

- [x] **nginx CRL enforcement** (`warden/nginx/nginx.conf`)
  `ssl_crl /etc/nginx/certs/ca.crl;` blocks revoked clients at the
  TLS handshake (Mode A).
  > ⚠️ nginx does **not** hot-reload the CRL file automatically.
  > After any revocation, run `docker exec warden-proxy nginx -s reload`
  > (Docker Compose) or restart the nginx pod (Kubernetes).

- [x] **Correct header forwarding** (`warden/nginx/nginx.conf` → `warden/mtls.py`)
  nginx sets `X-Client-Cert-Subject` (`$ssl_client_s_dn`) and
  `X-Client-Cert-Verify` (`$ssl_client_verify`).
  `MTLSMiddleware` reads these headers, parses the `CN=` field, and
  enforces the `MTLS_ALLOWED_CNS` allowlist.

- [x] **Mode B (direct uvicorn TLS)**
  Documented in `docs/deployment-guide.md §11.3`.
  `MTLSMiddleware` reads the peer cert from `request.scope["ssl"]`.
  Note: Mode B enforces only the CN allowlist — no CRL check.

- [x] **Key / cert file permissions**
  `gen_certs.sh` sets `chmod 600` on all `.key` files and
  `chmod 644` on all `.crt` and `.crl` files at generation time.

- [x] **Exempt paths from mTLS**
  `/health` and `/metrics` bypass `MTLSMiddleware` unconditionally
  (Kubernetes liveness/readiness probes + Prometheus scrape).

## 3. Scalability & Multi-Tenancy

- [x] **Token-bucket rate limiting via Lua**
  `warden/cache.py` runs an atomic Lua script inside Redis
  (`evalsha`) — no TOCTOU race conditions.
  Per-tenant limit is read from the `WARDEN_API_KEYS_PATH` JSON file
  and carried in `AuthResult.rate_limit`.

- [x] **Tenant isolation**
  Each API key in `WARDEN_API_KEYS_PATH` maps to a `tenant_id` and an
  independent `rate_limit`. Per-tenant `BrainSemanticGuard` corpora are
  stored in `_tenant_guards` dict in `warden/main.py`.

- [x] **WebSocket `/ws/stream`**
  Token-by-token streaming filter endpoint.
  Auth via `?key=<api_key>` query parameter.
  Emits `filter_result` events with `allowed`, `risk_level`, and
  `filtered_content` fields matching `FilterResponse` schema.

- [x] **Content-hash Redis cache** (`warden/cache.py`)
  SHA-256 of `(tenant_id, content)` → cached `FilterResponse`.
  5-minute TTL (configurable via `CACHE_TTL_SECONDS`).
  Fail-open: cache errors never block the filter pipeline.

## 4. GDPR Compliance

- [x] **Zero PII / Zero content logging**
  `data/logs.json` stores only: timestamp, risk level, request length,
  processing latency, flag types.
  Prompt text and secret values are **never** written to disk.

- [x] **Art. 15 — Right of Access** (`POST /gdpr/export`)
  Returns stored metadata for a single event by `request_id`.
  Requires `X-API-Key`. Returns 404 if no record found.
  ```json
  {"request_id": "...", "entry": {"ts": "...", "risk_level": "low", ...}}
  ```

- [x] **Art. 17 — Right to Erasure** (`POST /gdpr/purge`)
  Deletes all log entries with timestamp `< before` (ISO-8601).
  Returns count of removed entries.
  ```json
  {"removed": 142, "before": "2025-01-01T00:00:00Z"}
  ```

- [x] **Automated retention**
  Background task runs at startup and nightly.
  Configurable via `GDPR_LOG_RETENTION_DAYS` (default: 30).

## 5. Operational Readiness

- [x] **Admin UI** (`admin/app.py`, port **8502**)
  Streamlit panel: tenant management, dynamic rule CRUD,
  real-time event log viewer, system health cards.

- [x] **Prometheus metrics** (`GET /metrics`)
  Instrumented via `prometheus-fastapi-instrumentator`.
  Exposes request counts, p50/p95/p99 latency, and blocked-attack rate.
  Pre-built Grafana dashboard at `grafana/dashboards/warden_overview.json`.

- [x] **Helm chart** (`helm/shadow-warden/`)
  `helm test` runs 5 functional checks against the warden service:
  1. `GET /health` → `"status":"ok"`
  2. `POST /filter` safe content → `"allowed":true`
  3. `POST /filter` jailbreak → `"allowed":false`
  4. `POST /filter/batch` → batch response
  5. `POST /filter` strict mode → MEDIUM blocked

- [x] **Kubernetes resource limits** (`helm/shadow-warden/values.yaml`)
  | Pod | CPU limit | Memory limit |
  |-----|-----------|--------------|
  | warden | 2 cores | **2 Gi** |
  | analytics | 500 m | 512 Mi |
  | admin | 250 m | 256 Mi |

- [x] **HPA + PDB**
  `hpa-warden.yaml`: scales 2–10 replicas at 70% CPU.
  `pdb-warden.yaml`: `minAvailable: 1` prevents full eviction.

- [x] **CI pipeline** (`.github/workflows/ci.yml`)
  3 jobs: `test` (Python 3.11 + 3.12 matrix), `lint` (ruff + mypy),
  `docker-build` (import smoke + `/health` runtime check).
  Coverage gate: ≥ 75% (`--cov-fail-under=75`).
