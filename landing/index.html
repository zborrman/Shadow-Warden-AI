Encrypted Vault

How the vault works

01
Community Created

Ed25519+X25519 keypair (kid=v1). Private key Fernet-encrypted in vault. Public key shared with members.

02
Clearance Assigned

Each level → 32-byte Clearance Level Key via HKDF-SHA256 from X25519 private key + kid + level label.

03
Entity Encrypted & Stored

Random CEK wraps AES-256-GCM payload. CEK wrapped with level key. Ciphertext→S3. Metadata→PostgreSQL.

04
Authorized Download

Pre-signed S3 URL via proxy. Client unwraps CEK with level key, decrypts locally. Server sees only ciphertext.

Key Rotation

Root Key Rollover

Background ARQ worker re-wraps every entity CEK to the new key in configurable batches. Idempotent. Zero downtime.

01 New keypair generated (kid v1→v2). New traffic uses v2 immediately.
02 Old key → ROTATION_ONLY. CEKs re-wrapped batch-by-batch with Redis checkpoint.
03 M-of-N admin sign-off triggers crypto_shred(). Old key gone permanently.

Emergency Access

Last-resort access to ROTATION_ONLY keys for legal hold, forensics, or incident response. Gated by M-of-N multi-sig.
