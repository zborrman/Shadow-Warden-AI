Feature Gate — Code Integration

The Feature Gate is enforced server-side. Tenants cannot bypass tier limits regardless of client-side manipulation.

from warden.billing.feature_gate import FeatureGate

# Check if a tenant's plan allows a feature
gate = FeatureGate.for_tier("pro")

# Boolean feature check
if gate.batch_filtering:
    allow_batch_request()

# Raises PermissionError if below minimum tier
gate.require("enterprise", "byok")   # raises if not enterprise

# Check quota capacity
gate.require_capacity("quota_req_per_month", 100_000)

# Full feature dict for API response
features = gate.as_dict()
# → {"batch_filtering": True, "custom_rules": True,
#    "byok": False, "quota_req_per_month": 100000, ...}

Referral Flywheel & Quota Management NEW v2.9

Referral Flywheel Architecture

The referral system is a growth flywheel: every tenant can generate a unique code, share it, and earn bonus request capacity when new tenants sign up with that code. Codes are stored in Redis with a 90-day TTL and an in-memory fallback for tests.

Referral Code Lifecycle

1
Generate — POST /billing/referral/generate → REF-XXXXXXXX (8 hex chars, uppercase)
2
Share — share_url = https://app.shadowwarden.ai/signup?ref=REF-XXXXXXXX
3
Redeem — new tenant POSTs /billing/referral/redeem with {code, new_tenant_id}
4
Award — referrer gets +500 (starter/individual) or +2,000 (pro) bonus requests · new tenant gets +500 welcome bonus

Referral Rules

  • One active code per tenant at a time
  • Code TTL: 90 days (auto-expires in Redis)
  • Each code is single-use (marked redeemed on first use)
  • Self-referral rejected: new_tenant_id == referrer → 422
  • Enterprise tenants cannot generate referral codes (referral_program: false)
  • Bonus requests stack (multiple referrals accumulate)
  • Bonus requests do NOT reset monthly — they persist until consumed

Redis Keys

warden:ref:code:{CODE}   → JSON meta (referrer, plan, created)  TTL 90d
warden:ref:bonus:{tid}  → integer (bonus_requests remaining)
warden:ref:count:{tid}  → integer (total referrals made)

Quota Middleware

The QuotaMiddleware is an ASGI middleware that intercepts every POST /filter and POST /filter/batch request and enforces per-tenant monthly limits before any security processing begins.

How Quota is Calculated

effective_limit = plan_quota + referral_bonus_requests

# Redis key (35-day TTL — covers month boundary):
warden:quota:req:{tenant_id}:{YYYY-MM}

# On each qualifying request:
INCR warden:quota:req:{tid}:{YYYY-MM}

# If count > effective_limit:
#   Pro/Enterprise (overage_enabled=True): pass with warning
#   Starter/Individual: return HTTP 429 immediately

Quota Response Headers

# On every /filter response:
X-Quota-Used:  74320
X-Quota-Limit: 104000
X-Quota-Pct:   71.5

# At 80%+ usage:
X-Quota-Warning: approaching-limit

# On hard 429 (Starter/Individual over limit):
HTTP/1.1 429 Too Many Requests
{
  "detail": "Monthly quota exceeded. Upgrade plan.",
  "used": 1001,
  "limit": 1000,
  "upgrade_url": "/billing/upgrade?plan=individual"
}
QUOTA_HARD_BLOCK env var: Set QUOTA_HARD_BLOCK=false to convert all 429 responses to soft-warnings for all tiers. Useful for staging environments where you don't want to block test traffic. Default: true.

Warden Nexus — Federated Threat Intelligence NEW v2.9

Warden Nexus is a federated threat intelligence layer that allows Shadow Warden deployments to share anonymized attack signals across the entire fleet using the STIX 2.1 standard. A Bayesian consensus gate ensures no single malicious node can pollute the shared threat model.

What Warden Nexus Does

  • Each participating deployment publishes anonymized STIX 2.1 bundles for BLOCK-level attacks
  • Nexus aggregates signals fleet-wide into a shared ThreatVault
  • Bayesian consensus gate: requires ≥3 corroborating reports before promoting a threat signature globally
  • WormGuard integration: worm/propagation patterns auto-submitted to Nexus
  • All shared data is stripped of tenant identifiers, user content, and PII before transmission

Privacy Guarantees

  • Zero content shared: only threat geometry (Betti numbers, hyperbolic distance, threat class) + attack category is transmitted
  • Differential privacy: k-anonymity k≥5 enforced before publishing
  • Opt-in: Nexus disabled by default. Requires explicit WARDEN_NEXUS_ENABLED=true
  • GDPR compliant: no personal data leaves the tenant's network
  • Enterprise-only: Nexus participation requires Enterprise tier (Pro tenants receive read-only access)

🏢 Your Shadow Warden

  • Detects BLOCK-level attack
  • Extracts threat geometry coordinates
  • Strips all tenant/content data
  • Publishes STIX 2.1 bundle to Nexus
  • Receives updated global ThreatVault
  • Hot-reloads new threat signatures

5 Colang Flow Routes

clean_flow — Standard LLM response. No restrictions.

Integration Code

from warden.integrations.nemo_bridge import GeometricThreatBridge
from nemoguardrails import LLMRails, RailsConfig

# Initialize bridge
bridge = GeometricThreatBridge(warden_url="http://warden:8001")
config = RailsConfig.from_path("./colang/")
rails  = LLMRails(config=config)

# Process request
async def secure_chat(user_input: str, tenant_id: str) -> str:
    threat_coords = await bridge.analyze(user_input, tenant_id=tenant_id)
    colang_ctx    = bridge.to_colang_context(threat_coords)
    response      = await rails.generate_async(
        messages=[{"role": "user", "content": user_input}],
        context=colang_ctx
    )
    return response["content"]

GDPR & EU AI Act Compliance

Shadow Warden v2.9 is designed for GDPR Art.35 (DPIA) compliance and addresses EU AI Act Article 9 (risk management) and Article 15 (accuracy, robustness, cybersecurity) requirements for high-risk AI systems.

GDPR Controls (All Tiers)

  • Raw prompt content is never stored or logged
  • Secret Redactor strips PII before any downstream processing
  • Masking Engine: Fernet-encrypted vault for PHI restoration
  • Evidence bundles contain only metadata (flags, geometry, timing)
  • GDPR DPIA documentation in docs/dpia.md
  • POST /gdpr/erasure — right to erasure (tenant data)
  • GET /gdpr/export — data portability (tenant config)

EU AI Act Controls (Enterprise Tier)

  • Art. 9 — Risk management system: 9-layer pipeline + ThreatVault + Evolution Engine
  • Art. 15 — Accuracy & robustness: F1 0.947 public benchmark; adversarial robustness via TDA + Obfuscation Decoder
  • Art. 9+15 flag: controlled via FeatureGate.eu_ai_act (Enterprise only)
  • Break Glass: emergency override log for human-in-the-loop requirements
  • NVIDIA NIM routing for certified AI model execution

Upgrade Path: v2.4 → v2.9

Breaking changes: none. All existing env vars remain unchanged. Add the new v2.9 variables (LEMONSQUEEZY_*, QUOTA_HARD_BLOCK, WARDEN_NEXUS_*) to enable new features. QuotaMiddleware is automatically active — set QUOTA_HARD_BLOCK=false in staging to avoid blocking test traffic during rollout.
  1. Pull latest: git pull origin main
  2. Add new env vars to .env (see above)
  3. Rebuild containers: docker compose up --build -d
  4. Verify quota middleware: curl -I /filter → check for X-Quota-Used header
  5. Verify billing API: curl /billing/tiers → should return 4 tier objects
  6. Optional: Set WARDEN_NEXUS_ENABLED=true and WARDEN_NEXUS_API_KEY for Enterprise accounts
