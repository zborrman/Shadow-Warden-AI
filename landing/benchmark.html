Section 5

Research Methodology

Transparent, reproducible, open to third-party verification. Every design decision is documented below.

Evaluation Protocol

Binary classification: allowed=false ≡ detected threat. Metrics:

  • F1 = 2·Precision·Recall / (Precision + Recall)
  • Macro-average: F1 computed per category then mean — no category dominates due to sample count imbalance
  • TNR (True Negative Rate) used for benign controls instead of F1
  • Confidence intervals computed via bootstrap resampling (1000 iterations)

Corpus Construction

Corpus design principles:

  • Hand-labelled by two independent reviewers — disagreements resolved by majority
  • Sampled from real production attack logs (anonymised) + OWASP test suite
  • Benign controls drawn from Stack Overflow, GitHub issues, real business queries
  • No sample reused from SEMANTIC_THRESHOLD tuning set
  • Version-locked: data/benchmark_corpus_v2.9.json

Latency Measurement

Protocol designed to eliminate measurement noise:

  • 200 warm-up requests discarded (model pre-loaded, JIT compiled)
  • 200 timed requests via in-process TestClient (no TCP/TLS overhead)
  • MiniLM model pre-warmed via FastAPI lifespan() before measurement
  • Redis replaced by in-memory limiter (REDIS_URL=memory://) to isolate pipeline
  • P-values via Python statistics.quantiles()
  • Hardware: 4 vCPU / 8 GB RAM, Ubuntu 24.04, CPU-only PyTorch

Anti-Leakage Controls

Measures taken to prevent data leakage from evaluation into model tuning:

  • Benchmark corpus committed to data/benchmark_corpus_v2.9.json before any threshold tuning
  • Evolution Engine (Claude Opus) disabled during benchmarking (ANTHROPIC_API_KEY="")
  • Dynamic rules file reset to empty before each run
  • Corpus samples never appear in data/evolution_dataset.jsonl

OWASP LLM Top 10 Mapping

Attack categories mapped to OWASP LLM Top 10 (2025 edition):

  • LLM01 — Prompt Injection, Jailbreak, Obfuscation
  • LLM02 — Insecure Output / Data Exfiltration
  • LLM04 — Model DoS / Resource Abuse
  • LLM06 — Sensitive Information Disclosure (PII, secrets, tool abuse)
  • LLM08 — Excessive Agency (tool injection — partial coverage)

Version Policy

Results are version-stamped and immutable after publication:

  • Re-run after every minor version bump (e.g., 2.8 → 2.9)
  • Results committed to results/benchmark_YYYYMMDD.json
  • Historic results retained for regression tracking
  • Any F1 regression >0.01 blocks release in CI
  • Public results signed with SHA-256 in release notes
Section 6

How Each Defense Layer Is Validated

Each of the 9 stages has dedicated unit tests in addition to end-to-end benchmark evaluation. Stage isolation is achieved by patching downstream stages.

Stage 1 — Topological Gatekeeper
TDA n-gram point cloud · β₀/β₁ Betti numbers
Tested with synthetically generated obfuscation payloads of varying noise_score. Algebraic fallback (no ripser) tested separately. Adaptive thresholds (code=0.65, natural=0.82) validated with 30 code snippets vs 30 natural language samples.
100% obfuscation detection
<2ms P99
0% FPR on code
Stage 2 — Obfuscation Decoder
base64 · hex · ROT13 · Caesar · homoglyphs · UUencode
Depth-3 recursive decoding tested with nested multi-layer encodings (e.g., base64 of ROT13 of hex). Each encoding scheme tested independently, then in 2- and 3-layer combinations. Zero-width character injection validated against Unicode CLDR list.
100% decode on corpus
depth-3 recursive max
Stage 3 — Secret Redactor
15 regex patterns · Shannon entropy scan
Each of 15 regex patterns tested against known-format tokens (OpenAI sk-proj-, Anthropic keys, AWS AKIA, Stripe sk_live_, JWTs, SSH blocks, credit cards via Luhn validator, SSNs, IBANs). Shannon entropy threshold (≥4.5 bits/char) validated against 200 random strings of varying entropy.
15 regex patterns
≥4.5 entropy threshold
Luhn card validation
Stage 4 — ThreatVault
50+ named attack signature families
Each signature family tested with 3+ positive examples and 2+ near-miss benign examples. OWASP LLM Top 10 coverage verified by mapping each family to at least one OWASP category. Evolution Engine hot-reload tested by injecting new signatures at runtime and verifying detection without restart.
50+ signatures
hot-reload without restart
Stage 5 — Semantic Rules
42+ intent-aware regex · compound escalation
Compound escalation tested by constructing prompts with exactly 1, 2, and 3 MEDIUM-scoring patterns. Verified that 3×MEDIUM → HIGH upgrade fires correctly. Intent-aware patterns tested with paraphrased versions of attacks that bypass ThreatVault exact signatures.
42+ patterns
3×MED → HIGH
Stage 6 — Hyperbolic Brain
MiniLM-L6-v2 · Poincaré ball · 70% cosine + 30% hyperbolic
Blended score validated: 100 known-attack embeddings compared against 100 benign embeddings. Poincaré ball projection stability tested by verifying |expmap_0(v)| < 1 for all inputs including edge-case vectors (norm → ∞). Adversarial suffix stripping validated with 15 common suffix patterns (e.g., "Sure! Here's how"). Hyperbolic weight (30%) tuned via grid search on held-out dev set.
384-dim embedding
0.72 threshold

Stage 7 — Causal Arbiter
Bayesian DAG · Pearl do-calculus · gray-zone 0.50–0.72
5-node DAG validated by constructing 20 synthetic evidence combinations spanning the gray-zone score range (0.50–0.72) and verifying P(HIGH_RISK|evidence) monotonicity. Backdoor adjustment tested by injecting known confounders. Causal stage is skipped for scores outside gray-zone — skip-path benchmarked at <0.1ms overhead.

Stage 8 — ERS + Shadow Ban
Redis O(1) sliding window · gaslight / delay / standard
ERS sliding window tested with 5 simulated attack sequences — verified score accumulation and decay. Three shadow ban strategies tested: gaslight (response is plausible-but-incorrect), delay (verified async 3s without blocking event loop), standard (silent block). Fail-open on Redis outage verified by patching Redis client to raise ConnectionError.

Stage 9 — Decision + Evolution Engine
Signal aggregation · Claude Opus async rule generation
Decision aggregation tested across all verdict combinations (ALLOW/REDACT/HIGH/BLOCK). Evolution Engine tested with ANTHROPIC_API_KEY="" (air-gap mode) — verified all 9 detection stages function without API key. Hot-reload of new rules validated by checking add_examples() corpus size before and after Evolution Engine trigger. Async rule generation verified not to block the filter response path.

Section 9

Reproducibility Guide

Run the full benchmark locally in under 5 minutes. No running stack, no API keys, no GPU required.

1

Install dependencies

CPU-only PyTorch to avoid CUDA pull. The benchmark runs without Redis or any external services.

pip install torch --index-url https://download.pytorch.org/whl/cpu
pip install -e ".[dev]"
pip install -r warden/requirements.txt
2

Set environment variables

These isolate the benchmark from any running production stack. Evolution Engine is disabled (ANTHROPIC_API_KEY="") to prevent corpus leakage.

export ANTHROPIC_API_KEY=""
export WARDEN_API_KEY=""
export REDIS_URL="memory://"
export LOGS_PATH="/tmp/warden_bench_logs.json"
export MODEL_CACHE_DIR="/tmp/warden_bench_models"
export SEMANTIC_THRESHOLD="0.72"
3

Run the benchmark

Full pipeline benchmark with JSON and Markdown export. First run downloads MiniLM (~23 MB).

# Standard run — prints F1/precision/recall + latency percentiles
python scripts/benchmark.py

# Export results to JSON
python scripts/benchmark.py --export benchmark_results.json

# Markdown table output (for CI reports)
python scripts/benchmark.py --markdown

# Run pytest integration tests separately
pytest warden/tests/ -v --tb=short -m "not adversarial and not slow"
4

Verify results

Expected output for v2.9. Variance within ±0.02 F1 is acceptable (MiniLM has minor float non-determinism on different CPU microarchitectures).

# Expected output:
F1 macro-avg: 0.947 (±0.013 CI)
Precision: 0.961
Recall: 0.934
FPR: 3.1%
P50: 4.1 ms
P99: 18.6 ms
Throughput: 240+ req/s
5

Run NeMo Bridge demo (optional)

Simulates all 5 threat classes through the GeometricThreatBridge. No NeMo installation required for simulation mode.

python scripts/nemo_demo.py --simulate

# With Claude Opus Colang synthesis:
python scripts/nemo_demo.py --simulate --evolve
ℹ Results may vary by ±0.01–0.02 F1 across different CPU architectures due to floating-point non-determinism in MiniLM inference. The benchmark script seeds Python's random module but cannot control PyTorch's internal BLAS calls. This is expected and documented behaviour.
